## 项目背景

在多线程高并发场景下，系统默认的 malloc/free 分配方式存在锁竞争和性能瓶颈。tcmalloc（Thread-Caching Malloc）是 Google 提出的一种解决方案，通过线程本地缓存和中心缓存的分层设计，大幅提升多线程下的内存分配性能。

这个项目是为了深入理解内存分配器的工作原理而做的学习型项目。

## 为什么要做内存池

1. **性能**：减少系统调用频率，降低锁竞争
2. **碎片控制**：通过固定大小的内存块管理减少内存碎片
3. **理解底层**：理解操作系统内存管理的基本原理

## 核心结构

```
ThreadCache（线程本地缓存）
    ↓ 从 CentralCache 批量获取
CentralCache（中心缓存，加锁）
    ↓ 从 PageCache 获取 Span
PageCache（页级缓存）
    ↓ 向系统申请内存
System（操作系统）
```

## 技术难点

### 1. 线程本地缓存设计
每个线程拥有独立缓存，避免频繁加锁。使用 TLS（Thread Local Storage）实现。

### 2. 内存对齐与大小类划分
合理的内存对齐策略平衡了内存利用率和分配效率。参考 tcmalloc 的分级对齐策略：
- [1, 128]：8 字节对齐
- [129, 1024]：16 字节对齐
- [1025, 8*1024]：128 字节对齐
- ...

### 3. 中心缓存的并发控制
当线程本地缓存不足时，需要从中心缓存获取内存块。中心缓存使用互斥锁保护，但通过一次获取多个内存块减少加锁频率。

### 4. 内存回收
当线程释放内存时，如果本地缓存过多，将部分内存归还中心缓存。中心缓存的 Span 可以进一步合并为更大的页，归还操作系统。

## 性能优化思路

- **局部性优化**：线程优先使用自己的缓存
- **批量传输**：一次获取/归还多个对象，减少锁操作
- **懒分配**：只有在需要时才从下层获取
- **内存对齐**：平衡利用率和分配速度

## 我的收获

这个项目让我从底层的角度理解了内存管理的复杂性：

1. **并发编程的挑战**：不加锁的性能很好，但加锁后性能可能下降 10 倍以上
2. **分层设计的力量**：ThreadCache → CentralCache → PageCache 的分层思想可以应用到很多领域
3. **性能与复杂度的权衡**：极致的性能优化往往带来极高的实现复杂度
4. **源码是最好的教材**：阅读 tcmalloc 源码比看任何教程都高效

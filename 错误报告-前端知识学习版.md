## 错误一：中概念标点符号不是普通符号

### 现象

执行 `npm run build` 时，Turbopack（Next.js 的构建引擎）报错：

```
./src/data/abilities.ts:59:40
Expected ',', got 'ident'
```

指向这一行：

```typescript
"我对 HR 的兴趣，来自于"人如何被理解、匹配、成长和激发"这个问题。",
```

### 原因

这一行里包含中文字形双引号 `""`（U+201C / U+201D）。问题在于：

- 字符串开头的双引号是 **ASCII 双引号**（U+0022），这是 TypeScript 的字符串定界符
- 中间的 `""` 是 **Unicode 字符**（U+201C / U+201D），对 TypeScript 来说是普通文本内容
- 但某些文本编辑器、终端、或文件系统在保存时，**自动把 Unicode 引号转换成了 ASCII 双引号**

如果被转换了，编译器看到的其实是：

```typescript
"我对 HR 的兴趣，来自于"人如何被理解、匹配、成长和激发"这个问题。",
```

编译器读到第一个 `"` 开始字符串，读到第二个 `"`（来自中文引号的那一半）就认为字符串结束了，然后发现 `人如何被理解` 是一段不知道是什么的代码，于是报 `Expected ','`。

### 学了什么

**知识点：字符串定界符与字符编码**

| 符号 | Unicode 码点 | 在 TypeScript 中的作用 |
|------|-------------|----------------------|
| `"` (ASCII) | U+0022 | 字符串定界符 |
| `"` (左中文引号) | U+201C | 普通字符，无特殊含义 |
| `"` (右中文引号) | U+201D | 普通字符，无特殊含义 |

关键区别：**ASCII 双引号是语法符号，中文双引号是文字符号。** 它们长得像，但在计算机里是完全不同的二进制编码。

### 解决方案

三种方法都可以：

1. **改用「」**（最常见）：TypeScript 不认识中文符号，随便放
2. **用反斜杠转义**：`"来自于\"人\"这个问题"` — 但中文文案里加反斜杠不好看
3. **用模板字符串**：`` `我对 HR 的兴趣，来自于"人"这个问题` `` — 反引号可以天然包含双引号

### 下次怎么避免

- 在 TypeScript/JavaScript 的字符串里，**优先用单引号 `''` 或反引号 `` `` **
- 如果需要包含中文引号，用「」或『』代替，或者用反引号包起来
- 如果编辑器有"智能引号自动替换"功能，在代码文件里关掉它

## 错误二：组件库版本迭代导致导出名变更

### 现象

```
Export Github doesn't exist in target module
Did you mean to import Gift?
```

### 原因

我写的是：

```typescript
import { Github, ExternalLink, Mail, BookOpen } from "lucide-react"
```

lucide-react 是一个开源的图标组件库，早期版本（v0.x）中，品牌图标（Brand icons）的命名规则是：

- `Github` → GitHub 图标
- `Gitlab` → GitLab 图标
- `Twitter` → Twitter 图标

但在 **v1.x 版本**（我们安装的是 `"lucide-react": "^1.14.0"`），lucide 团队做了一个 breaking change：

> **品牌图标全部重命名了。** 旧的 `Github` 被移除，取而代之的是 `GitHubIcon` 这类新命名模式。

这是前端生态中的常见现象——**开源库的 API 不是永恒的，major version 升级可能不兼容。**

`Did you mean to import Gift?` 这个提示也很经典：**Turbopack 的静态分析找不到 `Github`，就尝试给你最接近的拼写建议——但它也只能做字符相似度匹配，`Github` 和 `Gift` 只是字母有点像，实际毫无关系。**

### 学了什么

**知识点：tree-shaking 与静态导出分析**

当你在写 `import { Github } from "lucide-react"` 时，Turbopack/Webpack 会做两件事：

1. **检查包是否真的导出了 `Github`** — 如果不存在，立即报错，而不是运行时才崩溃（编译时安全）
2. **Tree-shaking**：只打包你实际导入的图标，而不是整个 lucide-react（几十万个 SVG）

所以构建时就能发现错误，而不是部署上线后页面白屏。

### 解决方案

找到 lucide-react 当前版本有哪些导出：

```bash
# 查看 node_modules 里的类型定义
grep "export declare" node_modules/lucide-react/dist/lucide-react.d.ts
```

我们最终的方案是改用 `Code` 图标代替 `Github`——功能和风格接近，且不依赖品牌图标命名。

### 下次怎么避免

- 安装新版本依赖后，**先查阅对应版本的 CHANGELOG 或迁移指南**
- 对于图标库，通常优先选**非品牌图标**（像 `Code`、`Code2`、`GitBranch`），因为它们改名概率低
- 如果非要品牌图标，锁定版本：`"lucide-react": "0.400.0"`（不带 `^`），就不会意外升级

## 错误三：环境变量与 Shell 的路径问题

### 现象

在 Git Bash 中运行：

```bash
npx create-next-app@latest zxy-hr-profile
```

报错：

```
'"node"' 不是内部或外部命令，也不是可运行的程序
```

### 原因

Windows 上有多个 shell（命令解释器），每个 shell 有自己独立的 `PATH` 环境变量：

| Shell | PATH 配置来源 | 能否找到 Node |
|-------|-------------|-------------|
| CMD | 系统环境变量 | 能 |
| PowerShell | 系统环境变量 + Profile | 能 |
| Git Bash | 基本继承 + /usr/bin 等 | **可能找不到** |
| WSL | Linux 独立环境 | 需要另外安装 |

Git Bash 是一个模拟 Linux 环境的 shell，它默认的 `PATH` **可能不包含 `C:\Program Files\nodejs`**。所以当你敲 `node` 或 `npx` 时，shell 找不到可执行文件。

### 学了什么

**知识点：PATH 环境变量与可执行文件查找**

当你敲 `node --version` 时，shell 做的事情：

1. 检查是不是 builtin 命令（比如 `cd`）
2. 检查当前目录有没有 `node.exe` 或 `node`（不带 .exe 也要查）
3. 遍历 `PATH` 中的每个目录，找 `node.exe`
4. 如果都找不到 → `command not found`

`PATH` 的典型值（Windows）：

```
C:\Program Files\nodejs\        ← Node.js 在这里
C:\Windows\system32\
C:\Users\geniu\AppData\Local\Microsoft\WindowsApps\
```

如果 Git Bash 的 `PATH` 里没有 `C:\Program Files\nodejs\`，那它永远找不到 node。

### 解决方案

```bash
# 方案 A：临时加入 PATH
export PATH="/c/Program Files/nodejs:$PATH"
node --version  # 现在可以了

# 方案 B：用绝对路径
/c/Program\ Files/nodejs/node.exe --version

# 方案 C：（Windows 最佳实践）在 Git Bash 的 .bashrc 中永久加入
echo 'export PATH="/c/Program Files/nodejs:$PATH"' >> ~/.bashrc
```

### 下次怎么避免

- 在 Windows 上用 Git Bash 开发 Node.js 项目，**第一件事就是确认 PATH**
- 项目 README 里注明：`export PATH="/c/Program Files/nodejs:$PATH"`（如果环境需要）
- 或者直接使用 **PowerShell** 或 **CMD** 运行 node，这些 shell 默认能找到 Node.js
- 最好的方案：装一个 **nvm-windows**（Node Version Manager），它自动管理 PATH

## 错误四：Tailwind v4 的配置范式迁移

### 现象

传统 Tailwind 项目根目录有 `tailwind.config.ts`，但 `create-next-app` 初始化后找不到这个文件。

### 原因

我们用的是 **Tailwind CSS v4**，这是 2025 年的最新版本，配置方式发生了根本性变化：

**Tailwind v3（传统方式）：**

```typescript
// tailwind.config.ts —— 独立的 JS/TS 配置文件
export default {
  theme: {
    extend: {
      colors: {
        primary: "#0052d9",
      },
    },
  },
}
```

**Tailwind v4（新方式）：**

```css
/* globals.css —— 直接在 CSS 里用 @theme 定义 */
@theme inline {
  --color-primary: #0052d9;
  --color-primary-light: #e6f0ff;
  --color-primary-dark: #003a99;
}
```

### 学了什么

**知识点：CSS 自定义属性（Custom Properties,  aka CSS 变量）**

Tailwind v4 的配置底层原理是 **CSS 自定义属性**（也叫 CSS 变量）。

以前你在 JS 配置里定义：

```javascript
colors: { primary: "#0052d9" }
```

Tailwind 会在构建时转换成：

```css
.text-primary { color: #0052d9; }
.bg-primary { background: #0052d9; }
/* ...几十个 utility class */
```

现在你直接写 CSS 变量：

```css
@theme inline {
  --color-primary: #0052d9;
  /* Tailwind 自动生成:
     text-primary → color: var(--color-primary)
     bg-primary → background: var(--color-primary)
     border-primary → border-color: var(--color-primary)
     ... */
}
```

**为什么 Tailwind 团队要这么做？**

1. **零配置启动**：不需要 `tailwind.config.ts` 这个额外文件
2. **CSS 原生**：CSS 变量可以直接在浏览器 DevTools 里调试
3. **运行时兼容**：CSS 变量可以在 JavaScript 里读取（`getComputedStyle()`），JS 配置不行
4. **更细粒度**：你想覆盖某个 utility class 的行为直接在 CSS 里改，不用翻配置文件

### 解决方案

所有主题配置都在 `src/app/globals.css` 的 `@theme inline {}` 块里：

```css
@theme inline {
  --color-primary: #0052d9;
  --color-primary-light: #e6f0ff;
  --color-surface: #f5f7fa;
  --font-sans: var(--font-geist-sans);
}
```

然后在组件里直接用 Tailwind utility class：

```tsx
<button className="bg-primary text-white">按钮</button>
```

Tailwind v4 会自动解析 `--color-primary` 生成 `bg-primary`、`text-primary` 等 class。

### 下次怎么避免

- 用 `create-next-app` 初始化时，默认用 Tailwind v4。**不需要手动创建 `tailwind.config.ts`**
- 想自定义主题 → 改 `globals.css` 里的 `@theme` 块
- 想查看 Tailwind 生成的 class → 在组件上写 `bg-primary`，然后元素上右键 Inspect，看 DevTools 里有没有生效
- Tailwind v4 文档关键词搜索：`@theme`、`css-first configuration`

## 错误五：端口冲突与 Next.js 自动兜底

### 现象

启动 `npm run dev` 后，访问 `http://localhost:3000` 打不开，实际上服务器跑在 `http://localhost:3002`。

### 原因

Next.js 默认开发端口是 **3000**。但如果 3000 端口已经被占用了（比如另一个 Node 进程、或者其他应用），Next.js 不会报错退出，而是 **自动寻找下一个可用端口**。

这是 Next.js 内置的"优雅兜底"机制：

```
Port 3000 is in use, trying 3001...
Port 3001 is in use, trying 3002...
Ready on http://localhost:3002
```

### 学了什么

**知识点：TCP 端口与端口占用**

端口是计算机上"哪个程序负责接收哪种请求"的编号：

- `3000` → 你的第一个 Next.js 开发服务器
- `3001` → 被其他进程占用了（或者 Next.js 自动尝试的下一个）
- `3002` → 最终可用的端口

Windows 上查看端口占用：

```bash
netstat -ano | findstr :3000
# 输出示例：
# TCP    0.0.0.0:3000    0.0.0.0:0    LISTENING    12345
# 最后一列 12345 是进程 PID
```

### 解决方案

两个选择：

```bash
# 选项 A：杀死占用 3000 端口的进程
netstat -ano | findstr :3000   # 找到 PID
taskkill /PID 12345 /F          # 杀死进程

# 选项 B：指定端口
npm run dev -- -p 3001
```

### 下次怎么避免

- 启动开发服务器后，**看一眼终端输出的端口号**，别默认认为是 3000
- 如果经常端口冲突，在 `package.json` 里固定端口：
  ```json
  "scripts": {
    "dev": "next dev -p 3000"
  }
  ```

## 总结：前端项目从零到构建成功的完整心智模型

```
[你的代码] → [TypeScript 编译器] → [Turbopack 打包] → [静态 HTML 文件]
                                        |
                                  [运行时浏览器加载]
```

开发过程中遇到的错误，本质上都是这 4 个环节中的**信息不一致**：

| 环节 | 错误示例 | 本质问题 |
|------|---------|---------|
| **写代码时** | 中文引号被当成 ASCII | 你以为你在写文本，编译器以为你在写语法 |
| **导入依赖时** | `Github` 不存在 | 你记得的 API 版本不等于安装的版本 |
| **运行工具时** | `node: command not found` | 你所在的 shell 不知道 node 在哪 |
| **配置文件时** | 找不到 `tailwind.config.ts` | 你学过的教程版本不等于当前安装的版本 |

**前端开发最重要的三个习惯：**

1. **永远别信记忆，要看实际安装的版本** — `npm list lucide-react` 看看实际装了什么
2. **错误信息是线索，不是宣判** — 仔细读第一行报错（文件路径 + 行号 + 列号），然后看它希望是什么
3. **"不是 bug，是版本差"** — 前端生态变化极快，遇到 API 对不上先怀疑版本差异

这个项目用到的技术栈版本：

```
Next.js    16.2.6
React      19.2.4
TypeScript 5.x
Tailwind   4.x
lucide-react 1.14.0
```

> 以上知识足够你理解这个项目遇到的所有问题。未来学前端时，遇到错误就记住：
> **第一反应不是"我写错了"，而是"我了解的信息和实际环境哪里不一致"。**
